# Ключевые решения (ADR)

## Принято

1. **Supabase + Vercel free-тиры** — владелец не хочет платить; масштаб (5 юзеров) укладывается.
   Компромисс: пауза БД после недели простоя; Hobby-план Vercel формально некоммерческий.
2. **Всё через Server Actions, без API-роутов** — меньше кода, типобезопасно, RLS работает
   через куки-сессию пользователя автоматически.
3. **Права только на уровне БД (RLS), не в UI** — невозможно «забыть» фильтр в новой странице;
   миграция 0002. SECURITY DEFINER хелперы, чтобы политики не рекурсировали
   (классическая ловушка profiles→profiles).
4. **Регистрация по одноразовым инвайт-кодам** (таблица invite_codes + триггер на auth.users),
   а не открытая регистрация — CRM с клиентскими данными; и не ручное создание — владелец
   просил самообслуживание. Роль зашита в код приглашения.
5. **Личные задачи = tasks.project_id IS NULL** — без отдельной таблицы; RLS показывает их
   только автору/исполнителю (и админу).
6. **Деплой на Vercel полным каталогом** — основной путь через MCP `deploy_to_vercel`; при
   недоступности MCP допустим Vercel CLI, связанный с теми же project/team ID. Git-репозиторий к
   Vercel пока не подключён. Публичные `NEXT_PUBLIC_*` теперь хранятся как production env проекта
   Vercel, а не в коммитах или одноразовом `.env.production` payload.
7. **Сотрудники заведены как auth-заглушки, затем получили временные пароли** (не инвайтами),
   потому что на их profile.id уже ссылаются projects.responsible_id. Пароли назначены 2026-07-19,
   email подтверждены; сами пароли в Git не хранятся.
8. **Демо-данные удалены, реальные july-проекты** — источник истины: скриншот владельца;
   Notion — только для поля «ответственный» (владелец предупредил, что Notion устарел).
9. **Названия ролей**: owner (админ/владелец), pm, specialist, viewer — enum user_role из 0001.

## Пробовали и отказались

- **Локальный E2E без разрешённого сетевого доступа** — базовая песочница блокирует
  `*.supabase.co` и `*.vercel.app`. В сессии 2026-07-19 разрешённый egress использован только для
  целевых Auth/REST и production-проверок; полноценного постоянного E2E-набора пока нет.
- **`isDueToday()` внутри today-sort** — использовала реальную дату вместо параметра `today`,
  тесты упали при смене суток. Заменено на прямое сравнение `due_date === today`.
- **Embed `responsible:profiles(...)` без хинта** — падал PGRST201 (двусмысленность из-за
  project_members). Всегда писать `profiles!projects_responsible_id_fkey`.
- **seed-скрипт с SUPABASE_SERVICE_ROLE_KEY** (scripts/seed.mjs) — оставлен для локалки,
  но прод сидился напрямую SQL-ом через MCP (сервис-ключ в песочницу не передавали).
- **MCP Supabase/Vercel в сессии 2026-07-19 был недоступен** — миграция уже оказалась применена
  предыдущей попыткой и была подтверждена SQL-проверками; внешний деплой выполнен через Vercel
  CLI, а Supabase QA — через авторизованный Dashboard и публичные Auth/REST API.

## Ограничения, о которых помнить

- MCP-коннекторы (Supabase/Vercel/Notion) в сессиях нестабильны — флапают; критичные SQL
  сохранять во временный файл до применения. Разрешённый fallback должен использовать те же
  project/team ID и обязательно завершаться проверкой production.
- Обсидиан-хранилища у ассистента нет — «память проекта» живёт в repo (CLAUDE.md, docs/*)
  + заметка-дубль отдаётся файлом для ручного переноса в Obsidian.
- Ветка разработки: `claude/agency-ops-mvp-design-ykrrn0`; в main не мержилось, PR нет.
